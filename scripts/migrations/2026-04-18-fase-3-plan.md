## 1) Migrar `mediciones.ciclo_id` a `NOT NULL` de forma segura

**Prerrequisito duro**: `SELECT COUNT(*) FROM mediciones WHERE ciclo_id IS NULL = 0`.

Pasos:
1. Verificar en Neon: `SELECT COUNT(*) FROM mediciones WHERE ciclo_id IS NULL;` → debe ser 0.
2. Bloquear inserciones con `ciclo_id IS NULL`:
   ```sql
   ALTER TABLE mediciones ADD CONSTRAINT ck_mediciones_ciclo_required CHECK (ciclo_id IS NOT NULL) NOT VALID;
   ALTER TABLE mediciones VALIDATE CONSTRAINT ck_mediciones_ciclo_required;
   ```
3. Promover a NOT NULL y retirar CHECK redundante:
   ```sql
   ALTER TABLE mediciones ALTER COLUMN ciclo_id SET NOT NULL;
   ALTER TABLE mediciones DROP CONSTRAINT ck_mediciones_ciclo_required;
   ```
4. Reflejar en `ensureSchemaV2`: `INTEGER NOT NULL REFERENCES …` solo después del paso 3.

Rollback: `ALTER COLUMN ciclo_id DROP NOT NULL`. Reversible.

## 2) Condiciones antes de retirar `mediciones.compostera`

Checklist de bloqueos:
1. `ciclo_id` ya debe ser `NOT NULL` (punto 1).
2. **Ninguna lectura filtrando por `m.compostera`**: reescribir `getMediciones`, `getMedicionesExport` para JOIN vía `ciclos.compostera_id`. Reescribir `getMedicionesBySitio` / `getMedicionesExportBySitio` para JOIN vía `ciclos ci ON ci.id = m.ciclo_id` + `composteras c ON c.id = ci.compostera_id`.
3. **Ninguna escritura a `mediciones.compostera`**: simplificar `insertMedicion` y `updateMedicion`.
4. **Validación y tipos**: `validarMedicionInput` deja de exigir `compostera`. `MedicionInput.compostera` opcional. `Medicion.compostera` pasa a proyección derivada.
5. **Endpoints**: `/api/mediciones` GET `?compostera=N` vía `ciclos.compostera_id = N`. Idem export.
6. **Frontend**: `app/page.tsx` puede dejar de mandar `compostera`. `app/historial/page.tsx` sigue igual.
7. **CSV export**: columna "Compostera" desde `ciclos.compostera_id`.
8. **Backfill**: `ensureSchemaV2` backfill debe ser condicional o moverse a script separado.
9. **`consultas.compostera`**: queda fuera de esta fase.
10. **`ensureTable` v1**: puede eliminarse por completo.

Solo cuando 2–10 estén hechos y desplegados, se puede `DROP COLUMN compostera`.

## 3) Constraints / checks recomendados a nivel BD

Agregar en sub-fase 3a:

| Constraint | Qué protege | Reversible |
|---|---|---|
| `UNIQUE(id, compostera_id)` en `ciclos` | Habilita FK compuesta desde mediciones | Sí |
| `FOREIGN KEY (compostera, ciclo_id) REFERENCES ciclos(compostera_id, id)` en `mediciones` | **Garantía BD-level** de coherencia ciclo↔compostera | Sí |
| `CHECK (temperatura >= 0 AND temperatura <= 100)` | Defensa en profundidad | Sí |
| `CHECK (ph >= 0 AND ph <= 14)` | Idem | Sí |
| `CHECK (humedad >= 0 AND humedad <= 100)` | Idem | Sí |
| `CHECK (estado IN ('good','warning','danger'))` | Estados inválidos | Sí |
| `CHECK (dia IS NULL OR dia >= 0)` | Frontend podría enviar negativos | Sí |

Al drop de `mediciones.compostera` (3b) la FK compuesta también se va. Queda solo la FK simple a `ciclos(id)`.

Triggers **no** recomendados — prefiero FK compuesta mientras exista la columna, y eliminación de la columna después.

## 5) Estrategia incremental

### Sub-fase 3a — Hardening (reversible)
Script SQL `2026-XX-XX-v3-hardening.sql`:
1. Verifica precondiciones (huérfanas = 0, incoherencias = 0) con `DO $$ IF …`.
2. `ADD UNIQUE(id, compostera_id)` en `ciclos`.
3. `ADD FK compuesta` en `mediciones` → `ciclos(compostera_id, id)`.
4. `ALTER COLUMN ciclo_id SET NOT NULL` vía `ADD CHECK NOT VALID` + `VALIDATE` + `SET NOT NULL`.
5. Añadir CHECKs de rango (temp, ph, humedad, estado, dia).

Código de aplicación: ninguna dependencia fuerte. Los CHECKs aceptan datos actuales.

Rollback: cada `ADD CONSTRAINT` tiene su `DROP CONSTRAINT` inverso.

### Sub-fase 3b — Retiro de lectura/escritura de `mediciones.compostera` (código-only)
Commits de código (sin SQL):
1. Reescribir getters para JOIN vía ciclos.
2. Exponer `compostera` como `ci.compostera_id AS compostera` en SELECTs.
3. Quitar `compostera` del INSERT/UPDATE. Mantener parámetro ignorado.
4. Dejar de exigir `compostera` en `validarMedicionInput`.
5. Smoke test producción.

Columna sigue en BD. Rollback: revertir código.

### Sub-fase 3c — Drop de columna + limpieza (destructiva)
Script SQL `2026-XX-XX-v4-drop-medicion-compostera.sql`:
1. Verificar que ningún código lee/escribe `m.compostera`.
2. `DROP CONSTRAINT <fk_compuesta>;`
3. `DROP COLUMN compostera;`
4. (Opcional) `DROP UNIQUE(id, compostera_id)` en ciclos.

Código:
1. Eliminar `ensureTable()`.
2. Eliminar backfill de `ensureSchemaV2`.
3. Limpiar tipos.

**Sin rollback viable.** Última sub-fase.

SELECT COUNT(*) FROM mediciones WHERE ciclo_id IS NULL;
-- Incoherencias (debe ser 0)
SELECT COUNT(*) FROM mediciones m
  JOIN ciclos c ON c.id = m.ciclo_id
  WHERE c.compostera_id <> m.compostera;
-- Ciclo inexistente (debe ser 0)
SELECT COUNT(*) FROM mediciones m
  LEFT JOIN ciclos c ON c.id = m.ciclo_id
  WHERE m.ciclo_id IS NOT NULL AND c.id IS NULL;
-- Valores fuera de rango (debe ser 0)
SELECT COUNT(*) FROM mediciones
  WHERE temperatura < 0 OR temperatura > 100
     OR ph < 0 OR ph > 14
     OR humedad < 0 OR humedad > 100
     OR estado NOT IN ('good','warning','danger');

SELECT compostera_id, COUNT(*) FROM ciclos WHERE estado = 'activo' GROUP BY 1 HAVING COUNT(*) > 1;
